### Challenges

**1. Story 2.4 `Status` field not updated to `done` (repeating Epic 1 pattern)**

Amelia (Developer): "Story 2.4 still shows `Status: review` in the story file, even though sprint-status.yaml tracks it as done. This is the same discrepancy we flagged in Epic 1's retrospective."

Charlie (Senior Dev): "We committed to fixing this in the Epic 1 retro and it happened again. That tells us the process improvement wasn't embedded — it was aspirational."

Alice (Product Owner): "The root cause is likely that the story file update is a manual step that happens after the PR merges, and there's no forcing function. The developer is on to the next story."

Amelia (Developer): "The fix is automation, not more reminders. The story file status update needs to be part of the story completion checklist, not an afterthought."

**2. `#![allow(dead_code)]` is a blanket suppression that could mask real issues**

Charlie (Senior Dev): "Four of the five modules used `#![allow(dead_code)]` at the module level because their callers don't exist yet. That's the right short-term fix, but it means any genuinely dead code introduced by accident would be invisible to clippy."

Elena (Junior Dev): "Is there a better pattern? Like allowing at the item level instead of module level?"

Charlie (Senior Dev): "Yes — `#[allow(dead_code)]` on individual functions is more surgical. But it's more tedious when you have 8+ public functions. For Epic 3, when these modules get wired to callers, the allows should be removed entirely."

Amelia (Developer): "Adding that to the Epic 3 preparation: remove `#![allow(dead_code)]` from all Epic 2 modules once their callers are implemented in Epic 3."

**3. Story 2.3's `Default` trait gap between serde and `#[derive]`**

Elena (Junior Dev): "The checkpoint module had an interesting bug in the first test run — `#[derive(Default)]` generated a `machine_status = ""` (empty string), not `"active"`, because the serde default function wasn't connected to the Rust `Default` trait. The fix was a manual `Default` implementation."

Charlie (Senior Dev): "That's a well-known serde footgun. When you have `#[serde(default = "some_fn")]` and also `#[derive(Default)]`, serde's default applies to deserialization, Rust's `Default` applies to `Default::default()` — they're independent."

Amelia (Developer): "It was caught by the first test run, not in production, which is the right place to catch it. But we should document this footgun somewhere so future agents don't hit it again."

**4. `with_retry` in Story 2.5 had `last_err.unwrap()` in the exhausted-attempts path**

Charlie (Senior Dev): "The retry wrapper seeded `last_err` as `None` and then called `last_err.unwrap()` in the fallthrough path. That's a guaranteed panic if `max_attempts == 0`, and technically a violation of the no-unwrap contract even if `max_attempts` is always 3."

Dana (QA Engineer): "The code review caught it. The fix was to seed `last_err` with a synthetic fallback error. But this is the kind of subtle issue that's hard to catch without adversarial review — it looks correct at a glance."

### Key Insights

1. **Adversarial code review is essential for security-critical paths.** The OAuth token race window (Story 2.1), the non-atomic checkpoint save (Story 2.3), and the `parse_iso8601_utc` integer underflow (Story 2.3) were all caught only in review. None of these would have appeared as test failures — they're concurrency and edge-case issues that require human analysis.

2. **Cross-story learning propagated effectively when stories referenced previous story artifacts explicitly.** The `#![allow(dead_code)]` pattern, the `cargo clippy --all-targets` invocation, the unique temp file name scheme — all of these were documented in Story 2.3's Dev Agent Record and explicitly referenced by Stories 2.4 and 2.5. This is the pattern to maintain.

3. **Process agreements without enforcement mechanisms repeat as violations.** The story file `Status` field inconsistency appeared in Epic 1, was flagged in the retro, committed to as an action item, and appeared again in Epic 2 (Story 2.4). The lesson: behavioral agreements without tooling enforcement don't stick. Epic 3's Dev Notes should include the status update as a blocking checklist item.

4. **The no-new-crates constraint was a net positive.** The stdlib-only implementations for base64, ISO 8601 timestamps, and FNV-1a hashing added some implementation complexity but produced well-understood, auditable code with zero external supply chain exposure. This architectural decision was validated in practice.

5. **Module boundary enforcement was clean.** Zero instances of cross-module boundary violations (e.g., HTTP calls outside `github_api.rs`, checkpoint operations outside `checkpoint.rs`). The explicit "never does" columns in each story's architecture constraint table were effective guardrails.
