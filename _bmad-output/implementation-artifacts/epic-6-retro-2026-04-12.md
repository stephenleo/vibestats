## PART 1: EPIC REVIEW

### What Went Well

Amelia (Developer): "Epic 6 tackled real complexity — a multi-path Bash installer with 14 functions, Bash 3.2 compatibility, security requirements, and bats-core testing. Here's what stood out."

**1. The `_gh()` helper pattern was a major architectural win.**
Story 6.1 established the `_gh()` wrapper with a define-if-not-defined guard, allowing bats tests to pre-define their own stub before sourcing `install.sh`. This pattern was used consistently across all four stories and all test files. It made the entire installer testable without any shell binary mocking gymnastics. The `BASH_SOURCE` guard for `main()` complemented this perfectly — tests could source `install.sh` and call individual functions without running the full installer.

**2. Bash 3.2 compatibility discipline was maintained throughout.**
All four stories enforced the same constraint: no `declare -A`, no `mapfile`, no `[[ =~ ]]` regex captures, `command -v` instead of `which`, `case` statements instead of chained conditionals. The dev notes in each story carried explicit positive/negative examples. The implementation respected these constraints consistently. This is non-trivial discipline for a 14-function script that accumulates over four stories.

**3. Incremental function build-out over four stories worked cleanly.**
Stories 6.1 through 6.4 added functions to `install.sh` one layer at a time, with each story explicitly scoping what NOT to implement yet. Story 6.1 established the skeleton. Story 6.2 added the first-install path. Story 6.3 added the multi-machine path (and notably was developed in parallel with 6.2 after 6.1 merged). Story 6.4 added shared final steps. This layered approach meant no story tried to do too much, and each PR had a focused, reviewable scope.

**4. Security requirements were handled correctly throughout.**
The `VIBESTATS_TOKEN` piped-directly-to-gh-secret-set pattern (R-001, NFR7) was implemented correctly in Story 6.2 — never touching disk, never stored in a variable written to disk. `config.toml` was written with `chmod 600` immediately after creation in both Story 6.2 and 6.3 (NFR6). The security checklist in each story's PR checklist caught these requirements at implementation time, not at review time.

**5. Code review found real issues in each story.**
- Story 6.1: Missing default case in the checksum verification `case` statement — would have silently skipped checksum verification instead of exiting with an error.
- Story 6.2: Missing explicit error message when `gh auth token` fails in `store_machine_token`.
- Story 6.3: Redundant API calls — registry.json GET was called 3 times (existence check, `--jq .content`, `--jq .sha`), creating a race condition and unnecessary network round-trips. Fixed to parse content and SHA from a single GET response using python3.
- Story 6.4: Test design bug in ATDD test 7 (`$output` overwritten by a second `run` call before assertion).

All four reviews caught actionable defects. The review process is earning its keep.

**6. No `jq` dependency introduced.**
Multiple stories had complex JSON manipulation requirements (registry.json append with existing entry preservation, settings.json hook merge without clobber). All of these were implemented using Python3 stdlib (`import json, sys, base64`). This maintained the single runtime dependency contract: only `gh` CLI and Python3 (standard on target platforms) are required.

### Key Insights and Patterns

**Pattern 1: Define-if-not-defined guard is the correct pattern for testable shell helpers.**
The `_gh()` define-if-not-defined guard (checking `declare -f _gh` before defining) is a broadly applicable pattern for any shell script that needs testable external tool wrapping. It should be documented as a project convention for future shell work.

**Pattern 2: Python3 stdlib for JSON in Bash scripts is better than jq.**
When shell scripts need JSON manipulation (parsing, appending, merging), Python3 stdlib is more portable and more readable than `jq`. `jq` is not installed by default on macOS or many Linux distributions. Python3 is. This was the correct choice and should be the default approach for any future shell scripts that need JSON.

**Pattern 3: Security requirements need explicit test coverage, not just implementation notes.**
The `VIBESTATS_TOKEN` piped-never-to-disk security requirement was tested in Story 6.2 (`! grep -r "test-secret-token" "${HOME}" 2>/dev/null`). This negative assertion pattern is important: security properties need test coverage just like functional properties. Future security-sensitive code should include explicit negative tests for "token never on disk" type requirements.

**Pattern 4: The story-by-story regression test pattern is an effective mini-CI strategy.**
Running all prior stories' tests as part of each story's PR gate created a lightweight continuous integration effect within the epic. The test counts at each story (10 → 26 → 33 → 42) provide a clear signal of cumulative coverage growth. This pattern should be explicitly formalized in the team agreements for any future epic that builds on a single file across multiple stories.

**Pattern 5: Deferred review items need a "last epic" resolution policy.**
When a review item is deferred as "pre-existing — carry to next story," it needs to be resolved at some point. The EXIT trap item was deferred four consecutive times. The convention should be: if a deferred item reaches the last story of an epic, it becomes an explicit post-epic tech debt item rather than being deferred again.
