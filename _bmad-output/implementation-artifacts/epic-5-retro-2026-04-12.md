## PART 1: EPIC REVIEW

### What Went Well

Amelia (Developer): "Let me open with the wins. Epic 5 was Python and YAML, a different language surface than Epics 1–4. It shipped cleanly. Here's what stood out."

**1. Stdlib-only discipline held throughout.**
All Python scripts (aggregate.py, generate_svg.py, update_readme.py) used only standard library modules. No pip installs were needed in the Actions runtime. Story 5.1's dev notes were explicit: `json`, `datetime`, `pathlib`, `collections`, `sys`, `os` — and the implementation followed them exactly. Stories 5.3 and 5.4 maintained the same constraint without any exceptions.

**2. Fail-loudly contract applied consistently.**
Every Python script exits non-zero on failure. This is the opposite pattern from the Rust binary (which is silent by design). The contract was understood, documented in each story's dev notes, and implemented correctly. The GitHub Action surfaces failures and blocks partial commits — NFR13 was satisfied across all five stories.

**3. Code review caught a real security defect in Story 5.4.**
The Blind Hunter layer identified that `${{ inputs.profile-repo }}` was interpolated directly into a bash `run:` block, creating a script injection vector. The fix — passing the value via the step's `env:` block and using bash parameter expansion instead of command substitution — is the correct GitHub Security hardening mitigation. This is exactly the kind of issue code review is supposed to find, and it was caught before merge.

**4. TDD approach used correctly for YAML schema tests.**
Stories 5.4 and 5.5 both used a TDD pattern: pre-written tests with `@pytest.mark.skip` decorators, then activated in the green phase after implementation. This produced 25 schema tests for `action.yml` and 4 tests for `aggregate.yml`, covering all P0 acceptance criteria (no `continue-on-error`, no push triggers, correct step sequence, correct inputs).

**5. Dependency-graph kept current.**
Unlike Epic 3 (where the dependency graph drifted significantly), the dependency-graph.md was updated after every story batch in Epic 5. Multiple `chore:` commits tracked each merge. This made the "what's ready to work" state clear throughout the epic.

**6. Two-checkout pattern in action.yml was correctly reasoned.**
The `_profile_repo/` subdirectory pattern for the second checkout is subtle — the vibestats-data checkout must remain at `$GITHUB_WORKSPACE` root so `aggregate.py` can read `./machines/...` while the profile repo sits at `_profile_repo/`. The implementation got this right on the first attempt, as did the explicit path arguments passed to `generate_svg.py`.

### Key Insights and Patterns

**Pattern 1: Explicit CLI contracts in dev notes prevent ambiguity.**
Story 5.4's dev notes documented the exact CLI interface of each of the three Python scripts — argument names, defaults, what gets read, what gets written, and to which path. This prevented the kind of CWD confusion that would have caused aggregate.py to write `data.json` to the wrong location or generate_svg.py to use wrong default paths. The precision of these notes is what made the step sequence in action.yml correct on first implementation.

**Pattern 2: Security hardening belongs in code review, not in story tasks.**
The script injection defect in action.yml (Story 5.4) was not anticipated in the story's dev notes or anti-patterns section. It was found by the code review's Blind Hunter pass. This is correct — code review is the right place to catch security issues in YAML/shell interpolation. The process worked as designed.

**Pattern 3: `exit 0` vs. workflow termination in composite actions.**
Story 5.4 documented an important nuance: `exit 0` inside a composite action `run:` block exits only that step, not the whole workflow. Step 8 (push) still runs after Step 7 uses `exit 0` for the no-change early return. This is subtle and was documented explicitly in the dev notes. The team understanding of composite action step boundaries is now clear.

**Pattern 4: Test count as a health signal continues to be valuable.**
Cumulative test counts (21 after 5.1, 5+21 after 5.3, 80 total after 5.4) give a quick check that test coverage is growing. The gap in 5.2 is visible precisely because we track this.

## SIGNIFICANT DISCOVERIES ANALYSIS

Amelia (Developer): "No architectural assumptions were proven wrong in Epic 5. The GitHub Actions composite action pattern worked as designed. One discovery is worth documenting for Epic 8."

**Discovery 1: `concurrency:` group missing from aggregate.yml.**
The deferred-work note from Story 5.4 identifies that the push retry loop does not `git pull --rebase` between retries. If two vibestats workflows run against the same profile-repo concurrently, all retries will fail because the local branch is behind. The recommended mitigation is a `concurrency:` group on the `aggregate.yml` workflow. Story 5.5 did not add one.

**Impact on Epic 8:** Low — the story 5.5 template is for single-user profile repos where concurrent writes are rare. Story 8.3 (Marketplace publication) should document this limitation, and a follow-up update to `aggregate.yml` adding `concurrency: group: vibestats-${{ github.repository_owner }}` would be the correct fix. Not blocking.

**Discovery 2: No validation of empty --username in update_readme.py.**
The deferred-work note from Story 5.3 identifies that `argparse` accepts `--username ""`, which produces broken URLs. In the Actions context, `GITHUB_REPOSITORY_OWNER` is always set. No Epic 6 or 8 impact; the installer will pass the correct username.

No Epic 7 or 8 story redesign is required. All Epic 5 deliverables are sound.
