## Part 1: Epic Review

### Successes

**1. Clean, complete 100% delivery**

All 4 stories landed done with no carry-over. For a foundational epic where every downstream epic (2–8) is blocked on these artifacts, a clean 100% completion rate is exactly what was needed.

Alice (Product Owner): "The schemas doc is already paying dividends — I saw Epic 2 stories reference `docs/schemas.md` directly rather than having to re-derive the field names. That's the whole point."

Charlie (Senior Dev): "Story 1.2 was particularly clean. Zero warnings on `cargo clippy`, all 7 CLI invocations verified, `Cargo.lock` committed correctly as a binary project. Those details matter a lot later."

**2. Sandbox constraint handled gracefully (Story 1.3)**

The Astro initialization command required network access that wasn't available in the sandbox environment. Rather than blocking, the agent adapted: it manually scaffolded the Astro minimal template by fetching the upstream template files via the GitHub API. The build output was identical to the CLI path.

Dana (QA Engineer): "That's actually a great sign. When the happy path isn't available, the agent reasoned about an equivalent path and documented why. The completion notes explained it clearly."

**3. Security regressions caught early by code review (Story 1.1)**

The initial `.gitignore` rewrite dropped two important protections: (a) the `.env`/`*.toml.bak` secret-file entries, and (b) the `_bmad/`, `.claude/`, `.agents/` tooling directory entries. Both were caught in code review and patched before the story was marked done.

Charlie (Senior Dev): "This is exactly why we run adversarial review. A `.gitignore` that doesn't block `.env` files is a real security regression — not a style nit."

**4. Schema document is future-proof (Story 1.4)**

The `docs/schemas.md` went beyond minimum ACs. It includes design rationale (why Hive partitioning, why `harness=claude` enables multi-tool extensibility), anti-pattern warnings (`camelCase` vs `snake_case`, un-padded month/day), and cross-component impact notes identifying the 6+ downstream stories that depend on each schema element.

Alice (Product Owner): "The `harness=claude` partitioning note is smart product thinking embedded in a technical doc. Future Leo will thank past Leo."

**5. Cargo workspace isolation solved correctly (Story 1.2)**

A non-obvious problem: the worktree directory was nested inside the main repo, causing `cargo` to walk up and find the parent `Cargo.toml`. The agent identified the issue, applied the standard fix (`[workspace]` section), documented why, and moved on. No blocking issue created.

### Challenges

**1. `_redirects` over-matching (Story 1.3)**

The initial `site/public/_redirects` contained exactly `/:username  /u/index.html  200` as specified by AC#3. However, code review identified that this single catch-all rule incorrectly matches static assets (`/favicon.ico`, `/_astro/*`) and the dashboard shell itself (`/u`). The fix added explicit pass-through rules before the catch-all.

Root cause: The AC was written from the architecture spec, which specified the rule in isolation without considering Cloudflare Pages evaluation order. The code reviewer caught the gap.

Lesson: When a spec gives a single rewrite rule, evaluate whether the catch-all will intercept assets. Cloudflare Pages evaluates `_redirects` top-to-bottom; pass-throughs must precede catch-alls.

**2. Sandbox preventing npm/cargo execution required workarounds**

Story 1.3 needed `npm create astro@latest` (network), and Story 1.2 needed `cargo build` (file system writes to `/target`). Both required working within sandbox constraints, which meant manual scaffolding for Astro and careful path management for Cargo.

Root cause: Sandbox restrictions are intentional security boundaries. The workarounds were correct, but added implementation complexity.

Lesson: For project initialization stories in future projects, plan for possible sandbox limitations early and document the workaround approach in Dev Notes.

**3. Story 1.3 and 1.4 status discrepancy**

Both stories show `Status: review` in their files despite being tracked as `done` in `sprint-status.yaml`. The implementation artifacts were not updated to reflect final `done` status.

Root cause: Status sync between story files and sprint-status.yaml was inconsistent in Epic 1. Later epics show cleaner status management.

Lesson: Story file status should be updated to `done` as part of the story completion workflow, not just sprint-status.yaml.

### Key Insights

1. **Adversarial code review is essential for scaffolding stories.** Security regressions (`.gitignore` dropping secret-file protections) and functional regressions (`_redirects` over-matching) were both caught only in review. Scaffolding stories feel "low risk" but they set defaults that persist for the entire project.

2. **Dev Notes quality in Epic 1 set a high standard for downstream epics.** The detailed architectural rationale, explicit "do NOT do X" warnings, and cross-story dependency notes made implementation straightforward. This pattern should be maintained.

3. **Worktree-based development needs the `[workspace]` Cargo pattern documented once and referenced everywhere.** The cargo workspace traversal issue will recur in any Rust story run in a worktree. Story 1.2 solved it — that solution should be referenced in every subsequent Rust story's Dev Notes.
